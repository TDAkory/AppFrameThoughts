> [如何做 AI Agent 喜欢的基础软件](https://mp.weixin.qq.com/s/BZcRwgGZNinBK9K2L38LYg)

这篇文章由**TiDB创始人黄东旭**撰写，核心观点是：**基础软件（Infra）的主要使用者正从人类开发者迅速转向AI Agent**。为适应这一根本性转变，软件的设计理念需要进行系统性重构。

### 🧠 设计哲学：心智模型优先

当用户变为AI Agent时，软件暴露的核心不再是UI/API，而是其**背后稳定、古老且被广泛认知的心智模型**。

* **核心特征**：这些模型（如文件系统、SQL）经过几十年沉淀，**底层抽象极其稳定，上层组合非常灵活**。AI Agent通过学习海量人类代码，已将这些模式内化为强大的“先验知识”。
* **设计启示**：成功的软件应主动**顺应而非挑战AI已内化的认知结构**。AI Agent更擅长使用一个“它已经懂的系统”，并用极高的效率编写“胶水代码”来扩展它。
* **可扩展性**：优秀的心智模型必须是**可扩展的**。例如，**Linux VFS**或作者试验的**agfs**文件系统，允许在不改变“文件/目录”这一核心模型的前提下，接入全新的后端实现（如实现向量搜索）。

### 🗣️ 交互方式：为Agent设计的接口

一个好的系统接口需要同时满足三个条件，并在自然语言的灵活性与符号逻辑的确定性之间取得平衡：

| 核心要素 | 关键要求与说明 |
| :--- | :--- |
| **可被自然语言描述** | 系统的能力应本身适合用语言表达意图。放弃难以用语言描述的**图形界面**，转向编码或符号接口。 |
| **可被符号逻辑固化** | 必须在某个时刻将模糊的意图**收敛为无歧义的中间表示**（如SQL、脚本、代码），以便被存储、审计和复用。 |
| **能交付确定性结果** | 当前两点实现后，**确定性的结果是自然产物**。系统内部逻辑必须清晰、稳定。 |

作者认为，**代码是目前最好的逻辑符号描述**，因为它能用最少的Token（认知密度高）描述一个可无限重复执行的复杂过程。

### ⚙️ 系统特征：适应Agent的工作方式

为AI Agent服务的基础设施，其架构需要满足以下特征：

* **支持“日抛型”工作负载**：Agent擅长快速创建、试错、丢弃。因此，基础设施实例必须假设为**生命周期短、创建成本低、可抛弃的**，而不是“宝贵且需长期维护”的。
* **实现极致的低成本与高并发**：AI释放了长尾需求，可能面临管理**数百万个低频访问实例**的挑战。系统架构必须能支撑近乎零的边际成本与极高的并发度，传统的“一任务一进程”模型难以承受。
* **保障服务的连续性**：即使需求长尾、流量极低，**在线的服务也不应被随意暂停**（作者批评了为控成本暂停实例的做法）。可靠性与连续性需求并未降低。

### 💡 其他重要观点

* **开发范式**：作者不看好**LangChain**等较新的Agent框架，认为它们尚未形成稳定的心智模型。生态的重要性在于其背后的经典模型（如SQL），而非语法差异。
* **商业模式影响**：当代码生产能力因AI趋近于零时，**价值分配方式将发生根本变化**。软件需要服务更广泛、更碎片化的真实需求。（注：原文此处被截断，未完整展开）
